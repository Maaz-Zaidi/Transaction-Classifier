# Phase 12: Deep ML Analysis — Why the Model is Disappointing

**Date:** 2026-04-06

## Current State

73.9% accuracy (373/505) on the openai test set. 131 errors break down as:
- **86 raw finetune errors** (55.5% error rate on 155 predictions) — ML with no metadata
- **34 finetune_metadata errors** (22.2% error rate on 153 predictions) — ML with KB metadata
- **10 knowledge_base errors** (5.7% error rate on 175 predictions) — direct KB resolution
- **1 rules error** (7.7% error rate on 13 predictions)

The ML model is responsible for 120 of 131 errors. The pipeline works — direction detection and rules are near-perfect, KB direct resolution is 94.3% accurate. The ML is the bottleneck.

---

## Root Cause 1: The Training Dataset is a Template Machine with 847 Merchants

The mitulshah/transaction-categorization dataset has 3.6M rows but contains only **847 unique base merchant/concept names**. The entire dataset is generated by crossing those 847 names with:
- 5 countries (USA, UK, INDIA, CANADA, AUSTRALIA)
- 8+ time-of-day markers (MORNING, AFTERNOON, WEEKEND, RUSH HOUR...)
- 10+ location suffixes (STORE, ONLINE, BRANCH, CENTER, AIRPORT, CAMPUS...)
- Transaction codes (TXN123456)
- Payment methods (ACH, WIRE, CARD, CONTACTLESS, BANK TRANSFER...)

This template explosion creates the illusion of scale:

| Level | Count |
|-------|-------|
| Raw rows | 3,597,859 |
| Unique raw descriptions | 1,184,533 |
| Unique cleaned texts | 474,845 |
| **Unique base merchant names** | **847** |
| Average variants per base name | 560 |

Per-category, the base name counts are tiny:

| Category | Base Names | Rows |
|----------|-----------|------|
| Food & Dining | 62 | 358K |
| Transportation | 56 | 359K |
| Shopping & Retail | 51 | 360K |
| Entertainment & Recreation | 50 | 360K |
| Financial Services | 48 | 361K |
| Utilities & Services | 48 | 361K |
| Healthcare & Medical | 47 | 360K |
| Government & Legal | 46 | 361K |
| Charity & Donations | 41 | 360K |
| Income | 408 | 357K |

Income has 408 base names because it contains generic words (SALARY, PAYROLL, DIVIDEND, FREELANCE, etc.) that don't get stripped. The other categories average ~50 unique merchants each, padded to 360K rows by template expansion.

**Having 360K rows of SUBWAY variants doesn't teach the model anything that 100 rows wouldn't.** The model memorized a mapping of 847 names to categories. It did NOT learn semantic relationships between words and categories.

### What the model actually trained on

The models were trained on subsamples, not the full 3.6M:

| Model | Base Samples | After Augmentation | Unique Base Names |
|-------|-------------|-------------------|-------------------|
| finetune (baseline) | 8,000 | 8,000 | ~350 |
| finetune_augmented | 50,000 | 173,761 | ~477 |
| finetune_enriched | 20,000 | 55,750 | ~464 |

The enriched model (our best) trained on 55,750 samples covering approximately 464 of the 847 base names. The other ~383 names never appeared.

### 97% of test merchants are absent from training

Of the 505 openai test samples, only **15 (3.0%)** have an exact base-name match in training data. And 3 of those 15 matches are WRONG category (METRO→Transportation in training, Food in test; CREDIT→Income in training, Financial Services in test; CANADA→Income in training, Government in test).

The model has never seen: SHOPPERS DRUG MART, UBER EATS, LOBLAWS, CHARTWELLS, PRESTO, CIRCLE K, CANADIAN TIRE, T&T SUPERMARKET, FIDO, MTO, CINEPLEX, OSMOW'S, SHAWARMA PRINCE, BEAVERTAILS, FRESHCO, or any of the hundreds of Canadian-specific merchants in the test set.

### The training data looks nothing like real transactions

Training data looks like this:

```
SUBWAY - INDIA STORE
WALMART - CANADA - AFTERNOON
CREDIT UNION STORE CAMPUS
PAYROLL CENTER - DINNER TIME
TAX PREPARATION - AUSTRALIA STRIP MALL
```

Real Canadian bank transactions look like this:

```
CONTACTLESS INTERAC PURCHASE-4005 SHOPPERS DRUG M
UBER CANADA/UBEREATS
CHARTWELLS-OTTAWAUPARA
OCT-CONFEDERATION-RIDE
MACS CONV. STORE
```

### Vocabulary overlap confirms total domain mismatch:

| Metric | Value |
|--------|-------|
| Training vocabulary | 211,382 unique words |
| Test vocabulary | 842 unique words |
| **Vocabulary overlap** | **101 words (12.0% of test)** |
| **88% of test words** | **Never seen in training** |
| Base-name exact matches | 15/505 (3.0%), 3 of those are WRONG category |

### Training distribution mismatch:

| Category | Test % | Training % |
|----------|--------|------------|
| Food & Dining | **47.9%** | 10.0% |
| Shopping & Retail | **19.6%** | 10.0% |
| Transportation | 9.3% | 10.0% |
| Financial Services | 7.7% | 10.0% |
| All others | 15.5% | 60.0% |

The test set is 48% Food & Dining but training is perfectly balanced at 10% per class. The model has no prior that "most transactions are food" — it's equally confident about all 10 categories.

### What the model actually learned:

The training data teaches the model to memorize the 847 merchant names, not understand word semantics:

- **CREDIT** in training: "CREDIT UNION" (Financial Services, 7,940 samples) and standalone "CREDIT" (Income, 8,975 samples). Test has "CREDIT BALANCE" and "BONUS CREDIT" (both Financial Services) — model predicts Income because standalone "CREDIT" maps to Income.
- **UBER** in training: 6,889 samples, ALL Transportation. Test has "UBER EATS" (Food) — model has no way to know EATS modifies the category. "EATS" never appears in training.
- **METRO** in training: 7,070 samples, ALL Transportation (US transit systems). Test has "METRO" (Canadian grocery store) — model confidently predicts Transportation (0.704).
- **STORE** in training: 552,152 samples spread across ALL 10 categories equally (it's a template suffix). The word "STORE" carries zero category signal.
- **CONVENIENCE** in training: 0 samples. Never appears. So "MACS CONVENIENCE STORE" has no training signal for the key word.
- **SUPERMARKET** in training: 0 samples. "T&T SUPERMARKET" gets no help from the descriptor word.
- **PHARMACY** in training: 31,530 samples, 100% Healthcare. But "SHOPPERS DRUG" → "DRUG" has 0 training samples, "MART" appears only as a template suffix.

The model is a **lookup table for 847 US-centric merchant names** with template noise. It is NOT a semantic classifier. Having 3.6M rows instead of 8K doesn't help because there are only 847 things to learn — the augmented model (173K samples) scores the same as the baseline model (8K samples) on real data.

### Why more data from this dataset won't help:

Training on the full 3.6M rows would still only cover 847 base names. The finetune_augmented model already trained on 173,761 samples (covering ~477 base names) and performed no better than the 8K baseline on real data. The marginal return of seeing the 878th variant of "SUBWAY - INDIA STORE ONLINE" is zero. The dataset is information-exhausted at around 10-20K samples.

## Root Cause 2: The Foursquare Taxonomy Doesn't Match Our Categories

The KB enrichment pipeline feeds the model Foursquare place type labels. But the Foursquare taxonomy categorizes by **business type**, not by **transaction purpose**:

| Merchant | Foursquare Label | Our Category | Model Sees |
|----------|-----------------|--------------|------------|
| Shoppers Drug Mart | Retail > Pharmacy | Healthcare | "Retail > Pharmacy" → Shopping |
| Macs Convenience | Retail > Convenience Store | Food & Dining | "Retail > Convenience Store" → Shopping |
| T&T Supermarket | (no labels, only location) | Food & Dining | Nothing useful |
| Canadian Tire | Automotive Service > Repair Shop | Shopping | "Automotive Service" → Transportation |
| Circle K | (no labels, only location) | Food & Dining | Nothing useful |
| Loblaws | Landmarks > Garden (WRONG MATCH) | Food & Dining | Garden → Entertainment |

I tested the model on pure Foursquare labels with no merchant name:

```
"place types: Retail > Convenience Store"    → Shopping & Retail (0.680)  WRONG for food
"place types: Retail > Pharmacy"             → Shopping & Retail (0.579)  WRONG for healthcare  
"place types: Retail > Hardware Store"        → Shopping & Retail (0.737)  WRONG for Cdn Tire
"place types: Healthcare and Medicine > Pharmacy" → Healthcare (0.855)  CORRECT
"place types: Dining and Drinking > Restaurant"   → Food & Dining (0.770)  CORRECT
```

The model correctly maps "Dining and Drinking" → Food and "Healthcare and Medicine" → Healthcare. But Foursquare puts pharmacies under "Retail", convenience stores under "Retail", and grocery stores under "Retail > Food and Beverage Retail". The enrichment training (Phase 11) taught the model that "Retail > ..." means Shopping & Retail — which is correct for the Foursquare taxonomy but wrong for our category system.

**The Foursquare-to-our-category mapper (`map_foursquare_labels`) maps "Retail > Pharmacy" to Healthcare, but the model sees the raw label text "Retail > Pharmacy" and learns "Retail" = Shopping.**

This is a training data contamination: the enrichment pipeline feeds the model raw Foursquare labels, but the model should be learning our category mapping, not Foursquare's taxonomy structure.

## Root Cause 3: KB Retrieval Returns Wrong Merchants

Several metadata errors trace to the KB matching the wrong merchant:

| Query | KB Matched | Strategy | Problem |
|-------|-----------|----------|---------|
| LOBLAWS | Loblaws - Garden Centre | rerank | Matched a garden centre variant |
| CDN TIRE STORE | Q Stop - CDN Tire Gas | rerank | Matched a gas station, not the store |
| UBER CANADA/UBEREATS | Uber Canada HQ | rerank | Matched corporate office |
| FIDO MOBILE | Fido Mobile Go | rerank | Fido store is "Electronics Store" in Foursquare |
| T&T SUPERMARKET | T&T Supermarket | rerank | Entry has no labels, only location |
| Circle K | Circle K | exact | Entry has no labels, only location |
| CARLETON UNIVERSITY PA | Carleton University | exact | No category, no labels |

The "rerank" strategy matches are particularly bad — they find loosely related entries that have wrong metadata. The exact matches are often entries with no useful Foursquare labels at all (no mapped_category, no raw_category_labels), so the enrichment adds only a location string that provides no category signal.

## Root Cause 4: Financial Services is Unsolvable with This Data

12 of 39 Financial Services test samples are errors. The test samples include:

```
PAYMENT - THANK YOU / PAIEMENT - MERCI    → model predicts Income
BONUS CREDIT / CREDIT ADDITIONNEL         → model predicts Income  
CREDIT BALANCE                            → model predicts Income
OVERLIMIT FEE                             → model predicts Income
NOTICE                                    → model predicts Income
INTERAC-SC                                → model predicts Utilities
NSLSC                                     → model predicts Healthcare
CASH WITHDRAWAL BR TO BR                  → model predicts Income
NAT STU LN SER                            → model predicts Income
ULTIMATE CURREN                           → model predicts Entertainment
WU - C/O CASH S                           → model predicts Shopping
```

These are **bank statement notices and fees**, not merchant transactions. The training data has ZERO samples for PAYMENT, BALANCE, FEE, NOTICE, OVERLIMIT, NSLSC, or any bank statement language. Financial Services training is entirely bank names (TD BANK, CHASE, WELLS FARGO) and financial products (MORTGAGE, INVESTMENT, INSURANCE, TAX PREPARATION).

No amount of ML training on the current dataset can fix this. These words simply don't exist in the training vocabulary.

## Root Cause 5: The Model Architecture is Fine, the Input is the Problem

MiniLM with WordPiece tokenization handles the text correctly:

```
SHOPPERS DRUG MART  → [shop, ##pers, drug, mart]     (4 tokens)
UBER EATS           → [uber, eats]                    (2 tokens)
OVERLIMIT FEE       → [over, ##lim, ##it, fee]        (4 tokens)
CANADIAN TIRE       → [canadian, tire]                 (2 tokens)
```

The tokenization is fine — WordPiece breaks unknown words into meaningful subwords. The max_length of 96 tokens is sufficient (no truncation found in any test sample). The model architecture (classification head on CLS token) is standard and appropriate.

The model IS responsive to good metadata. When I feed it pure Foursquare labels:
- "Dining and Drinking > Restaurant" → Food & Dining (0.770) CORRECT
- "Travel and Transportation > Fuel Station" → Transportation (0.700) CORRECT
- "Healthcare and Medicine > Pharmacy" → Healthcare (0.855) CORRECT

The problem is not the model — it's what goes into it.

## What Can Be Done: Ranked by Impact

### 1. HIGHEST IMPACT: Fix the Foursquare Label → Category Signal Path (~+3-5%)

The enrichment training teaches the model raw Foursquare label text. But the model learns "Retail > X" = Shopping, which conflicts with our category mapping where "Retail > Pharmacy" = Healthcare and "Retail > Convenience Store" = Food.

Fix: Instead of feeding the model the raw Foursquare label, feed it **our mapped category name** as the metadata signal:

```
Current:  "SHOPPERS DRUG MART. place types: Retail > Pharmacy"
Better:   "SHOPPERS DRUG MART. category signal: Healthcare & Medical"
```

Or map at the label level before enrichment:
```
Current:  "place types: Retail > Pharmacy; Retail > Cosmetics Store"
Better:   "place types: Healthcare & Medical (pharmacy)"
```

This way the model learns our category system, not Foursquare's taxonomy. The enrichment training already has the correct category labels — it just needs to use them in the text format that the model sees.

### 3. HIGH IMPACT: Expand Rules for Bank Statement Patterns (~+2-3%)

12 Financial Services errors are bank statement notices. These follow patterns that rules can catch:

```yaml
- pattern: "PAYMENT.*THANK"    → Financial Services
- pattern: "CREDIT.*BALANCE"   → Financial Services  
- pattern: "OVERLIMIT"         → Financial Services
- pattern: "SERVICE.*FEE"      → Financial Services
- pattern: "ANNUAL.*FEE"       → Financial Services
- pattern: "INTEREST.*CHARGE"  → Financial Services
- pattern: "CASH.*WITHDRAWAL"  → Financial Services
- pattern: "CORRECTION"        → Financial Services
- pattern: "AUTHORIZATION.*EXPIRED" → Financial Services (or strip prefix)
```

These are structural patterns (bank statement language), not merchant names. They belong in rules, not ML.

### 4. MEDIUM IMPACT: Fix KB Retrieval for Key Merchants (~+2-3%)

Several errors trace to KB returning wrong entries:
- LOBLAWS matches "Loblaws - Garden Centre" instead of the grocery store
- CDN TIRE matches "Q Stop - CDN Tire Gas" instead of Canadian Tire
- Multiple entries have no Foursquare labels (T&T, Circle K, Carleton University)

Fixes:
- Prefer exact canonical name matches over partial rerank matches
- When multiple KB entries match, prefer the one with labels/category over one with only location
- Add manual category mappings for high-frequency Canadian merchants that have no Foursquare labels

### 5. HIGH IMPACT: Generate Diverse Semantic Training Data (~+5-10%)

The core problem: 847 base merchant names is not enough to learn semantic word-category relationships. The model needs to learn that words like SHAWARMA, GRILL, BAKERY, PIZZA mean Food — not by memorizing specific restaurants, but by seeing enough diverse examples that the word itself becomes the signal.

Approach: Generate thousands of realistic merchant names that teach semantic associations:
- Food descriptors: "[RANDOM NAME] SHAWARMA", "[NAME] GRILL", "[NAME] BAKERY", "[NAME] PIZZA", "[NAME] SUSHI", "[NAME] CAFE"
- Shopping descriptors: "[NAME] ELECTRONICS", "[NAME] CLOTHING", "[NAME] FURNITURE"
- Service descriptors: "[NAME] MOBILE", "[NAME] WIRELESS", "[NAME] INTERNET"
- Use actual Canadian merchant patterns: truncated names, no spaces, bilingual text

This is fundamentally different from the mitulshah approach (847 fixed names × template noise). Instead, generate diverse names where the CATEGORY-RELEVANT WORD varies and the brand name is random. Train the model to weight descriptor tokens over brand tokens.

Combine with real Foursquare KB data: the 1.8M KB entries contain real merchant names with real categories. Use a subset of KB entries with verified categories as additional training data (the KB lookup already confirmed the category, so we're not leaking test data).

### 6. MEDIUM IMPACT: Better Tokenization of Enriched Text

The current enrichment format is verbose:
```
SHOPPERS DRUG MART. merchant identity: Shoppers Drug Mart. external metadata: place types: Retail > Pharmacy; Retail > Cosmetics Store; Retail > Miscellaneous. place types: Retail > Pharmacy; Retail > Cosmetics Store
```

This wastes tokens on duplicate information. A cleaner format:
```
SHOPPERS DRUG MART [Healthcare & Medical] pharmacy, cosmetics
```

Shorter enrichment means more of the 96-token budget goes to the actual merchant name.

## The No-Go Approaches

**Training on more of the mitulshah dataset**: The finetune_augmented model already trained on 173K samples and performed no better on real data than the 8K baseline. The dataset has 3.6M rows but only 847 unique concepts. Adding more rows from the same 847 names teaches nothing new. The dataset is information-exhausted.

**Better base model**: MiniLM is fine. The problem is training data, not model capacity. Switching to a larger model (DeBERTa, BERT-large) would not help because the vocabulary gap stays the same — the model can't learn SHOPPERS DRUG MART regardless of how many parameters it has.

**Better public dataset**: There is no publicly available Canadian bank transaction dataset. The mitulshah dataset is the best available synthetic option. Kaggle alternatives (apoorvwatsky/bank-transaction-data, USA Banking Transactions Dataset) are either US-focused, too small, or for fraud detection. No alternative solves the fundamental problem of missing Canadian merchant names.

**Zero-shot / LLM classification**: An LLM could classify these correctly because it has world knowledge (knows SHOPPERS DRUG MART is a pharmacy). But latency and cost make this impractical for a real-time pipeline. The zero-shot fallback already exists in the pipeline but has its own accuracy issues.

**Using openai test data for training**: The 505-row openai set is our only labeled real data. Splitting it (e.g., 70/30 train/test) would give ~350 training samples but leave only ~150 for evaluation, too small for reliable accuracy measurement. And 350 samples doesn't solve the core issue — we'd teach the model 350 specific merchants but it still wouldn't generalize to new ones.

---

## Summary

The ML model is not broken — it was set up to fail. The mitulshah dataset has 3.6M rows but only **847 unique merchant names** — the rest is template noise. The model memorized a lookup table of 847 US-centric names. 97% of test merchants don't appear in that table. The Foursquare metadata pipeline partially compensates (+8.5% on the metadata path) but introduces its own errors through taxonomy mismatches and wrong KB matches.

**The fundamental blocker is training data diversity, not quantity.** More rows from the same 847 names don't help. The model needs to learn semantic word-category relationships (SHAWARMA→Food, GRILL→Food, MOBILE→Utilities) from diverse examples, not memorize a fixed merchant list. The highest-leverage changes are: (1) fix the Foursquare label mapping so metadata actually helps, (2) expand rules for structural patterns the ML can't learn, (3) generate semantically diverse training data that teaches word-category associations, (4) leverage the 1.8M KB entries as a training data source.
